# 多代理协作写作系统：实现与示例输出

## 1. 实现概述

本项目使用 LangGraph 编排研究、撰写、审核和润色四个专业代理，并通过独立的
FastMCP HTTP 服务提供 DuckDuckGo 搜索与提示词。所有主代理和备用代理默认使用
OpenAI `gpt-5`。

共享的 `AgentState` 保存主题、风格、目标字数、研究报告、初稿、审核建议、终稿、
用户补充信息、过程日志和结构化重试事件。每个节点只返回自己新增的状态字段，由
LangGraph 合并后传递给下一个节点。

## 2. 执行流程

```text
用户输入主题、风格、字数
        ↓
Research Agent（MCP 网络搜索）
        ↓
Writing Agent
        ↓
Review Agent
        ↓
Polishing Agent
        ↓
终稿 + 全过程记录 + 异常处理日志
```

每个阶段发生异常时执行以下恢复策略：

1. 主代理首次失败后最多重试两次。
2. 三次均失败则切换至使用高级提示词的备用代理。
3. 备用代理仍失败则请求用户补充信息，并保留之前的完整状态再次执行。

## 3. 示例输出

以下内容展示程序输出文档的结构。实际运行时研究内容、文章和日志由模型动态生成。

### 最终文章：AI Agent 如何改变知识工作

AI Agent 正在让人工智能从“回答问题的工具”转变为“能够完成任务的协作者”。
传统聊天机器人通常等待用户提出问题，再生成一次回复；AI Agent 则可以围绕目标
进行规划、查找资料、调用工具并检查结果。例如，在撰写行业报告时，它可以先搜索
公开信息，再整理关键事实、生成初稿并交由审核代理检查。

这种能力通常由大语言模型、工具调用、记忆和任务编排共同实现。大语言模型负责理解
意图与生成内容，搜索或数据库工具提供外部信息，状态管理保存中间结果，而工作流框架
负责确定下一步执行哪个节点。多代理系统进一步将复杂任务拆分给不同角色，使研究、
写作和审核能够各自使用更有针对性的提示词与质量标准。

AI Agent 能够提升知识工作的效率，但它并不天然可靠。搜索失败、来源质量不高、
模型幻觉和外部 API 中断都可能影响结果。因此，实际系统需要保留来源、记录中间过程，
并在失败时采用重试、备用代理和人工补充信息等恢复措施。人类仍应对高风险结论和最终
发布内容负责。

未来的 AI Agent 会更加注重可观测性和协作能力。相比单纯追求更长的回答，稳定的工具
使用、明确的状态流转和可审计的决策记录，才是它们进入真实业务流程的关键。

- 风格：通俗易懂

- 模型：gpt-5

#### 研究报告

- 核心概念：AI Agent 是能够围绕目标进行推理、调用工具并执行多步任务的系统。
- 关键组件：大语言模型、工具调用、状态或记忆、任务编排。
- 应用：资料研究、内容生产、客户服务和流程自动化。
- 风险：幻觉、来源错误、外部服务故障和缺少人工监督。
- 趋势：多代理分工、可观测性、可靠恢复和人在回路。
- 参考来源：实际运行时由 DuckDuckGo 搜索结果提供，不允许模型编造 URL。

## 4. 验证

`tests/test_retry.py` 使用模拟代理覆盖以下路径，无需调用真实 OpenAI API：

- 主代理重试后恢复
- 主代理失败后切换备用代理
- 请求用户补充信息后恢复
- 三级恢复全部失败并抛出明确异常
